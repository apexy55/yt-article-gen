# yt-article-gen

YouTube 字幕转中文文章生成器，基于 Cloudflare Workers + Gemini AI。

## 项目信息
- GitHub 仓库地址：https://github.com/apexy55/yt-article-gen
- 部署后的公开访问网址：https://yt-article-gen.apexytech55.workers.dev/

## 功能特性

- 输入 YouTube 视频链接，自动获取字幕
- 调用 Gemini 2.5 Flash API，**流式输出**生成中文文章（一边生成一边展示）
- 支持自然语言**生成要求**（风格、受众、约束条件等）
- 每个章节内置 **[5W1H]** 按鈕，点击可生成结构化 Who/What/When/Where/Why/How 总结
- 5W1H 请求使用服务端保存的上下文，无需重新提交文章
- 字幕获取失败时自动回退至内置示例字幕

## 技术架构

- **运行时**: Cloudflare Workers (Edge)
- **语言**: TypeScript
- **AI**: Google Gemini 2.5 Flash (Free API)
- **前端**: 原生 HTML/CSS/JS，内联于 Worker 返回

## 部署指南

### 1. 克隆仓库

```bash
git clone https://github.com/apexy55/yt-article-gen.git
cd yt-article-gen
npm install
```

### 2. 设置 Gemini API Key

前往 [Google AI Studio](https://aistudio.google.com/api-keys) 获取免费 API Key，然后：

```bash
npx wrangler secret put GEMINI_API_KEY
# 输入你的 API Key 并回车
```

### 3. 本地开发

```bash
npm run dev
```

### 4. 部署到 Cloudflare Workers

```bash
npm run deploy
```

部署后获得公开访问 URL，格式如：
`https://yt-article-gen.<your-subdomain>.workers.dev`

## 说明文档
本项目是一个基于 Cloudflare Workers 的 YouTube 内容文章生成器。用户输入一个带字幕的 YouTube 视频链接后，系统会自动获取字幕内容，调用 Gemini 生成中文结构化文章，并通过流式输出的方式实时渲染到页面中。 在文章生成后，用户还可以针对任意章节点击 [5W1H] 按钮，获取该章节的结构化总结。

### 如何获取和处理 YouTube 字幕
在 YouTube 字幕获取部分，后端首先会解析用户输入的 URL，提取 videoId。最初的思路是直接请求 YouTube 页面并解析字幕轨道，再把字幕事件整理成 transcript 文本；但在实际调试过程中，这条链路会受到页面结构变化、字幕轨道可用性以及访问稳定性的影响，因此中间又引入了 Supadata (第三方字幕数据服务) 来提升字幕获取的成功率和稳定性。
项目实现采用了“原始抓取字幕思路 + 第三方字幕数据服务 + fallback 字幕”的方案。 当直接从 YouTube 页面提取字幕不够稳定时，Supadata 提供了一个更直接的数据获取路径，用来帮助拿到视频对应的字幕或转写结果，再统一整理为带时间戳的 transcript 文本，供后续 Gemini 生成文章使用。 
这一点其实也是项目中比较重要的一次工程调整：题目表面上要求的是“获取字幕并生成文章”，但真正落到实现层面，难点往往不在文章生成本身，而在于 字幕来源是否稳定。 因此后面把 Supadata 纳入方案，用它来解决 YouTube 字幕抓取不稳定的问题；如果 Supadata 或实时抓取仍然失败，再回退到内置示例字幕，保证整个演示流程始终可运行。

### 如何调用 Gemini 并实现流式输出
项目在 src/gemini.ts 中封装了与 Gemini 的交互逻辑，由后端将“字幕内容 + 用户生成要求 + 固定格式约束”组合成提示词 prompt 后发送给 Gemini 模型。 文章生成使用的是流式接口，因此返回内容会以分段的形式持续到达，而不是一次性完整返回。
在实际调试过程中，Gemini API 相关问题是项目里反复处理最多的一部分。首先遇到的是模型版本兼容问题，早期使用的模型标识无法正常被当前接口识别，因此需要根据实际可用模型及时调整到可用的 Gemini 模型名称，确保请求链路能够跑通。 其次，线上环境里多次出现配额与限流问题，典型的 429 quota exceeded 场景； 从工程角度看，这一部分的真正的难点往往不只是“能调用”，而是“在不断变化的模型版本、权限和额度条件下仍能稳定工作”。
Cloudflare Worker 会持续读取 Gemini 返回的流式数据，并把新增内容实时转发到前端；前端收到后立即追加到文章展示区域中，因此用户可以看到内容边生成边显示的效果。 这种方式满足了流式输出 streaming output 的要求。

### 如何根据用户生成要求影响输出结果
前端提供了“生成要求（可选）”输入框，用户可以直接用自然语言输入要求，例如希望文章更长、更正式、面向初学者、强调商业分析，或者要求章节更多、解释更细。 后端会把这些自然语言要求附加到提示词 prompt 中，与字幕正文一起提发送给 Gemini，从而影响最终生成结果的篇幅、风格和结构。

### 如何实现章节级 5W1H 总结
章节级 5W1H 总结是本项目的另一个重点，也是多次迭代的部分。最关键的要求是：用户点击某一章旁边的 [5W1H] 按钮时，不能由前端把整篇文章再次提交给模型，而是要依赖服务端之前保存的生成上下文。 为此，项目在生成文章时，会把本次会话相关的字幕、文章内容、用户要求等信息保存在服务端上下文中，后续点击 5W1H 时，前端只需要传当前章节信息和会话标识，后端再结合已保存的全文上下文来完成总结生成。 这样做既满足题目要求，也让 5W1H 的结果更准确，因为它不是孤立地看某一段文字，而是同时参考了整篇文章和整段视频的背景。
5W1H 这部分在调试时也经历了多次修正。一个常见问题是，如果只把当前章节文本单独发给模型，它容易生成过于泛化、脱离全文语境的答案；因此后续实现里加入了“章节内容 + 全文上下文 + 视频字幕背景”的组合方式，让模型在总结当前章节时能保留足够的全局信息。 另一个问题是，如果 5W1H 的输出格式不够严格，前端渲染时会出现结构不稳定，因此在接口设计上会尽量要求模型输出结构化字段，减少自由格式文本带来的不确定性。 从工程实现上看，这部分体现了一个很典型的经验：大模型功能要想“可用”，往往需要通过上下文设计、接口约束和多轮 prompt 调整，把原本开放式的生成行为逐步收敛成可预测、可渲染、可复用的结构化能力

### 主要工程取舍和亮点
从工程取舍上看，项目有几个比较明确的设计决定。第一，采用“真实字幕抓取 + fallback 字幕”的双保险方案，兼顾真实性和稳定性；真实抓取证明系统具备完整链路能力，而 fallback 机制保证了在外部依赖不稳定时产品仍可演示。 第二，项目没有把所有逻辑写在一个文件里，而是拆分为路由入口、前端页面、字幕处理、Gemini 调用、上下文存储等多个模块，使代码结构更清晰，也便于快速排查是字幕、模型、前端渲染还是状态保存出了问题。
这个项目的主要亮点并不只是“接上了一个大模型”，而是在有限时间里完整打通了从 YouTube 字幕获取、到 Gemini 流式生成、到前端实时渲染、再到章节级 5W1H 总结的整条链路。 更重要的是，在 Gemini API 配额、模型版本、输出篇幅控制，以及 5W1H 上下文设计这些实际问题上，项目都经过了多轮调试和修正，最终形成了一个可运行、可展示、可继续扩展的版本。
